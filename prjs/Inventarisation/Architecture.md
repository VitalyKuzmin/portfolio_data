# Архитектура проекта "Инвентаризация объектов освещения"

## 1. Обзор системы
Программно-аппаратный комплекс для автоматизации сбора, учета и паспортизации объектов городского наружного освещения.
**Бизнес-задача:** Заменить ручной сбор данных (бумага) и оцифровку сторонними подрядчиками на собственный цифровой конвейер.

## 2. Компоненты системы

### A. Мобильный клиент (Data Collection)
*   **Роль:** Инструмент полевых бригад. Разрабатывался подрядчиком по моему ТЗ.
*   **Функции:**
    *   **Офлайн-режим:** Работа в зонах без связи, кэширование тайлов карты.
    *   **Геопозиционирование:** Автоматическое определение координат опоры (GPS/GLONASS).
    *   **Атрибутирование:** Формы ввода данных (тип опоры, светильника, состояние, фотофиксация).
*   **Синхронизация:** Пакетная отправка данных в центральную БД при появлении сети.

### B. Центральный узел (1С:Предприятие)
*   **Роль:** Master Data Management (MDM) и Backend.
*   **Функции:**
    *   Единое хранилище справочников оборудования и структуры города (Улица -> Участок -> Опора).
    *   Прием сырых данных с мобильных устройств.
    *   Предоставление API для клиентских приложений.
    *   Связь с бухгалтерским учетом активов.

### C. Рабочее место инженера (Electron Desktop App)
*   **Стек:** Electron, Node.js, JavaScript, HTML/CSS.
*   **Роль:** Основной инструмент для валидации данных, проектирования и аналитики. Разработан мной.
*   **Почему Electron?** Требовалась работа с веб-картами (JS), и возможность выполнения тяжелой логики на клиенте.

#### Ключевые модули Electron-клиента:

1.  **Картографический модуль (Yandex Maps API):**
    *   **Проблема:** Рендеринг десятков тысяч объектов (опор) в масштабах города "кладет" браузер.
    *   **Решение:** Реализована **кластеризация** (объединение точек в группы при отдалении). Использовался `ObjectManager` Яндекса для оптимизации производительности (JSON-based загрузка данных).

2.  **Модуль светотехнических расчетов:**
    *   **Функция:** Позволяет выбрать участок улицы, задать параметры светильников и рассчитать освещенность дорожного полотна.
    *   **Сложность:** Отображение результатов расчета (изолиний, градиентов освещенности) поверх карты.
    *   **Реализация:**
        *   Математический расчет (возможно, интеграция с C++ библиотекой или JS-реализация формул).
        *   Отрисовка векторных слоев (Canvas/SVG) поверх подложки карты с точной геопривязкой.

3.  **Модуль отчетности:**
    *   Генерация паспортов объектов и сводных ведомостей в PDF на основе данных из 1С и карты.

## 3. Технические вызовы и решения (System Design)

### Вызов 1: Синхронизация данных
*   **Проблема:** Как обеспечить консистентность данных между мобильным устройством, 1С и Electron-клиентом?
*   **Решение:** 1С выступает как "Source of Truth". Мобильное приложение работает в режиме "append/update". Electron-клиент вычитывает актуальное состояние. Использовались HTTP-сервисы 1С для обмена JSON-пакетами.

### Вызов 2: Производительность карты
*   **Проблема:** Отображение векторной графики (результатов расчета) и тысяч маркеров одновременно.
*   **Решение:** Разделение слоев. Маркеры кластеризуются. Слой освещенности рендерится только для активного (выбранного) участка, а не для всего города сразу. Использование Canvas для отрисовки сложных графиков вместо DOM-элементов.

## 4. Результат
Система позволила полностью отказаться от услуг внешних подрядчиков по инвентаризации. Время обработки данных сократилось с недель (ручной перенос с бумаги) до минут (синхронизация).
