# Процесс разработки и CI/CD: БЛ Био (MVP)

## 1. Философия
Для MVP-проекта был выбран максимально простой и быстрый процесс, ориентированный на скорость итераций, а не на сложную автоматизацию. Основная цель — быстрая проверка гипотез и получение обратной связи от заказчика.

## 2. Стек инструментов
- **Репозиторий:** GitLab.
- **CI/CD:** GitLab CI для базовой проверки бэкенда.
- **Окружение:** Одна виртуальная машина (VM) для Staging и Production.

## 3. Процесс разработки и сборки

### Backend (Python)
Был настроен минимально необходимый пайплайн в GitLab CI, который запускался при коммите в `main` ветку:
1.  **`lint`**: Проверка качества кода с помощью `ruff` и `black`.
2.  **`test`**: Запуск unit-тестов через `pytest`.
3.  **`build`**: Сборка Docker-образа и отправка его в GitLab Container Registry.

**Развертывание (Deployment):**
- Производилось **вручную**.
- Я подключался к серверу по SSH и выполнял команды `docker pull <image>` и `docker-compose up -d --force-recreate` для обновления сервиса. Для MVP этого было достаточно.

### Frontend (Angular Fork)
- Разработка велась локально.
- После внесения изменений я запускал команду `ng build` для сборки статических файлов.
- Собранные файлы (`dist/` папка) вручную копировались на сервер в директорию, которую раздавал Nginx.

## 4. Тестирование
- **Unit-тесты:** Покрывали только критически важную бизнес-логику на бэкенде.
- **Ручное E2E тестирование:** После каждого развертывания я лично или вместе с заказчиком проводил сквозное тестирование основного сценария: "Залогиниться -> Увидеть свои теплицы -> Изменить температуру -> Убедиться, что команда дошла до контроллера".

Этот упрощенный подход позволил максимально ускорить цикл "идея -> код -> фидбэк", что является ключевым для успешного MVP.
