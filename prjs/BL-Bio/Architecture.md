# Архитектура проекта "БЛ Био" (MVP)

## 1. Контекст и Бизнес-задача
**Задача:** Быстро создать облачную платформу для управления сетью "умных" теплиц.
**Ограничения:** Сжатые сроки, ограниченный бюджет.
**База:** Промышленные контроллеры Wiren Board (Linux, MQTT, Modbus) с open-source веб-интерфейсом (Angular).

**Проблема базового решения:**
Стандартный UI Wiren Board работает локально (один контроллер — один интерфейс). Нет разграничения прав, нет реестра теплиц, нет удаленного доступа "из коробки" без VPN/Static IP.

## 2. Архитектурное решение (Hybrid Cloud/Edge)

Мы выбрали гибридную модель:
*   **Edge (На объекте):** Контроллер управляет теплицей автономно (полив, свет). Логика не зависит от интернета.
*   **Cloud (Портал):** Управляет доступом, хранит список пользователей и привязку к контроллерам. Служит точкой входа.

## 3. Компоненты системы

### A. Cloud Backend (Python + PostgreSQL)
*   **Роль:** Панель администрирования и точка входа (Identity Provider).
*   **Стек:** Python (Django + DRF) или FastAPI. Выбрали **Python** для скорости разработки API.
*   **Функции:**
    *   **Реестр:** Хранение сущностей "Пользователь", "Теплица", "Контроллер", "Камера".
    *   **RBAC (Role Based Access Control):** Админ создает пользователей и назначает им теплицы. Пользователь видит только свои объекты.
    *   **Backup:** Хранение дампов конфигурации контроллеров (на случай выхода оборудования из строя).
    *   **Telegram Gateway:** Сервис рассылки уведомлений. Пользователь подписывается на бота, а бэкенд маршрутизирует алерты с конкретного контроллера нужному пользователю.

### B. Frontend (Angular + Wiren Board UI Fork)
*   **Решение:** Вместо написания UI с нуля, мы взяли open-source фронтенд Wiren Board.
*   **Модификация:**
    *   Обернули его в "Портал" (Dashboard со списком доступных теплиц).
    *   При выборе теплицы фронтенд переключает контекст (подключается к нужному MQTT-брокеру/WebSocket каналу).
    *   Скрыли системные настройки Linux, оставив только бизнес-функции (управление светом, климатом).

### C. Edge Device (Wiren Board)
*   **Роль:** Микро-сервер на объекте.
*   **Стек:** Linux (Debian), MQTT Broker (Mosquitto), wb-rules (JS-like движок правил).
*   **Автономность:** Все расписания и сценарии хранятся и исполняются локально. Если связь с облаком пропадает, теплица продолжает функционировать.
*   **Связь:** Контроллер выступает как MQTT-клиент или держит VPN-туннель до облака, чтобы принимать команды.

## 4. Оценка стека и решений (System Design Review)

*   **Python для Backend:** Для MVP — **оптимально**. Позволило быстро развернуть REST API и админку.
*   **Переиспользование Angular UI:** **Стратегически верно**. Сэкономило месяцы разработки сложных виджетов (ползунки, графики, интерактивные переключатели).
*   **Локальная логика (Edge):** **Критически важно**. В агро-секторе интернет нестабилен. Облачное управление должно быть только инструментом мониторинга и коррекции, а не "мозгом" системы.
*   **Отсутствие Big Data в MVP:** Решение хранить историю датчиков локально на контроллере (или не хранить долго) позволило сэкономить на дорогой инфраструктуре для Time Series данных на старте.

## 5. Итог
Проект был сдан в срок. Заказчик получил возможность продавать "Теплицу под ключ" с облачным управлением, не разрабатывая собственное железо и софт с нуля, а интегрируя готовые промышленные решения под своим брендом.
