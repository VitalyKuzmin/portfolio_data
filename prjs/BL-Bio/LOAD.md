# Оценка нагрузки и ресурсов: БЛ Био (MVP)

Для MVP-проекта расчет ведется для небольшого числа пилотных установок.

## 1. Исходные допущения

- **Количество теплиц (контроллеров):** 20
- **Частота отправки телеметрии:** 1 сообщение каждые 10 секунд с каждого контроллера.
- **Количество одновременных пользователей UI:** 5 (администраторы, наблюдающие за состоянием).

## 2. Расчет нагрузки

| Метрика | Расчет | Значение |
| :--- | :--- | :--- |
| **Сообщений MQTT в минуту** | 20 контроллеров * 6 сообщений/мин | **120 msg/min** |
| **Средний TPS (Transactions Per Second)** | 120 / 60 | **2 msg/s** |
| **Объем трафика MQTT** | 2 msg/s * ~200 байт/msg | **< 1 KB/s** |
| **WebSocket-соединений** | - | **5** |

## 3. Оценка ресурсов (MVP)

- **Инфраструктура:** Вся система (Python-бэкенд, MQTT-брокер Mosquitto, PostgreSQL) может без проблем работать на **одной небольшой виртуальной машине** (например, 2 CPU, 4 GB RAM) в Docker-контейнерах.
- **Нагрузка:** Для MVP-стадии нагрузка является незначительной.

## 4. Вопросы масштабирования (Post-MVP)

При росте до тысяч устройств текущая архитектура столкнется с проблемами:
1.  **MQTT-брокер:** Один инстанс Mosquitto не справится. Потребуется переход на кластеризуемый брокер (например, VerneMQ или EMQX).
2.  **База данных:** Хранение time-series данных телеметрии в PostgreSQL — плохая практика в больших масштабах. Потребуется миграция на специализированную Time-Series СУБД (например, TimescaleDB, InfluxDB или ClickHouse).

**Итог:** Для MVP нагрузка минимальна, но при планировании развития продукта необходимо закладывать смену ключевых компонентов стека данных.
