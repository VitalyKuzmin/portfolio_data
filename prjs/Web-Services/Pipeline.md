# Процесс разработки и CI/CD: Онлайн-калькуляторы

## 1. Философия
Независимое развертывание каждого микросервиса. Команда может обновлять сервис калькулятора несколько раз в день, не затрагивая основной монолит на Bitrix.

## 2. Стек инструментов
- **Репозиторий:** Polyrepo-подход. Каждый микросервис в своем Git-репозитории.
- **CI/CD:** GitLab CI.
- **Окружения:** Docker, Staging и Production на нескольких VM.
- **Реестр артефактов:** GitLab Container Registry.

## 3. Модель ветвления
- **`main`:** Ветка, которая всегда отражает состояние Staging-окружения.
- **Теги (e.g., `v1.2.3`):** Только тегированные коммиты могут быть развернуты в Production. Это обеспечивает версионирование и возможность отката.
- **`feature/*`:** Ветки для разработки.

## 4. CI/CD Пайплайн (для каждого микросервиса)

### Этап 1: `lint-and-format`
- Запуск `ruff` и `black` для проверки стиля и качества кода.

### Этап 2: `test`
- Запуск `pytest`.
- В GitLab CI используется `services` для поднятия временного контейнера с PostgreSQL, чтобы интеграционные тесты работали в изолированном окружении.

### Этап 3: `build`
- Сборка Docker-образа сервиса.
- Образ тегируется хешем коммита и `latest`, после чего пушится в GitLab Container Registry.

### Этап 4: `deploy-staging`
- Этот этап запускается автоматически при коммите в `main`.
- GitLab Runner по SSH подключается к Staging-серверу и выполняет `docker-compose pull && docker-compose up -d` для обновления сервиса.

### Этап 5: `deploy-production`
- **Запускается вручную** только при создании тега.
- Аналогично Staging, Runner подключается к Production-серверам и обновляет сервис.
- Используется простейший **Rolling Update**: поочередно обновляем контейнеры на нескольких нодах, чтобы избежать простоя.

## 5. Наблюдаемость (Observability)
- **Логирование:** Все сервисы писали логи в `stdout`, которые собирались через Filebeat и отправлялись в **Elasticsearch**. Просмотр и анализ — в **Kibana**.
- **Мониторинг:** Prometheus собирал метрики (RPS, latency, error rate) с каждого сервиса. Дашборды и алерты были настроены в **Grafana**.
