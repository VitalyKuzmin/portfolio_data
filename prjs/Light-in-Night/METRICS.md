# Метрики проекта: Light in Night Pro

Несмотря на то, что это desktop-приложение, мы внедрили сбор анонимной телеметрии для понимания использования продукта и его стабильности.

## 1. Продуктовые метрики (Вовлеченность и использование)

- **DAU / MAU:** Количество уникальных активных пользователей в день/месяц. Ключевой показатель "жизни" продукта.
- **Feature Adoption Rate:** Доля пользователей, которые использовали ключевые функции (например, "Расчет освещенности дорог", "CUDA-рендер", "Создание отчета по ГОСТ").
- **Session Length:** Средняя продолжительность сессии. Помогало понять, насколько глубоко пользователи работают с программой.
- **Error/Crash Rate:** Процент сессий, закончившихся ошибкой или "падением" приложения. Отправлялись анонимные крэш-репорты.

## 2. Метрики производительности и здоровья системы (Health Metrics)

Собирались для понимания, на каком "железе" работает наш софт и где есть проблемы с производительностью.
- **Average Calculation Time:** Среднее время выполнения светотехнического расчета.
- **Average Render Time:** Среднее время фотореалистичного рендера.
- **Hardware Statistics:** Анонимная статистика по конфигурациям пользователей (модели CPU/GPU, объем RAM). Помогало при планировании оптимизаций.
- **API Error Rate:** Процент ошибок при обращении клиента к серверным сервисам (лицензирование, синхронизация каталогов).

## 3. Метрики команды и процессов

- **DORA-метрики:**
    - **Deployment Frequency:** Частота выпуска новых версий (цель была — ежемесячные релизы с патчами по необходимости).
    - **Change Fail Rate:** Доля релизов, требующих срочного хотфикса.
    - **MTTR (Mean Time to Recovery):** Среднее время на исправление критичных багов, найденных на проде.
- **Code Review Time:** Среднее время "проживания" Merge Request'а до мержа.
- **Bus Factor:** Отслеживался через матрицу компетенций, чтобы избежать ситуации, когда только один человек разбирается в критичном модуле (например, в движке рендеринга).
