# Процесс тестирования: Light in Night Pro

## 1. Стратегия тестирования

Мы применяем многоуровневый подход, основанный на пирамиде тестирования, чтобы обеспечить качество и стабильность сложного desktop-приложения.

```mermaid
graph TD
    subgraph "Автоматизированное тестирование"
        A[Unit-тесты (GTest, Pytest)]
        B[Интеграционные тесты (DLL)]
        C[UI/E2E-тесты (Airtest)]
    end
    subgraph "Ручное тестирование"
        D[Регрессионное и исследовательское тестирование]
    end
    
    A --> B --> C --> D
```

## 2. Уровни тестирования

### Уровень 1: Unit-тестирование
- **Цель:** Проверка корректности отдельных модулей и функций в изоляции.
- **Инструменты:**
    - **C++ (Engines):** `GTest`. Особое внимание уделялось математическому ядру (`Calculation Engine`), которое имело близкое к 100% покрытие тестами.
    - **Python (Backend Services):** `Pytest`. Покрывалась вся бизнес-логика сервисов лицензирования, телеметрии и синхронизации.
- **Исполнение:** Запускаются автоматически в CI/CD пайплайне (Jenkins/GitLab CI) на каждый коммит.

### Уровень 2: Интеграционное тестирование
- **Цель:** Проверка корректности взаимодействия между ядром приложения (Core UI) и динамическими библиотеками (DLL-движками).
- **Примеры сценариев:**
    - Корректная загрузка и выгрузка `Calculation Engine`.
    - Передача данных из UI в `Graphic Engine` для отрисовки.
    - Вызов `Report Engine` с корректными параметрами и получение отчета.
- **Исполнение:** Полу-автоматизированные скрипты, запускаемые на сборочном стенде.

### Уровень 3: UI / E2E-тестирование
- **Цель:** Автоматизация ключевых пользовательских сценариев для отлова регрессий в UI.
- **Инструмент:** **Airtest** (фреймворк на базе Computer Vision).
- **Процесс:**
    1. На отдельном тестовом стенде развернуты виртуальные машины с Windows и Linux.
    2. После каждой успешной сборки `develop` ветки, скрипты Airtest запускали приложение.
    3. Выполнялся набор сценариев: "Создать проект дороги", "Добавить светильник", "Запустить расчет", "Сгенерировать отчет".
- **Результат:** Позволило значительно сократить время на ручной регресс и ловить баги на ранней стадии.

### Уровень 4: Ручное тестирование
- **Ответственный:** QA-инженер.
- **Процесс:**
    - **Тест-кейсы:** Ведение и актуализация тест-кейсов в системе управления тестированием.
    - **Регрессионное тестирование:** Полный прогон всех тест-кейсов перед каждым релизом.
    - **Исследовательское тестирование (Exploratory Testing):** Поиск нетривиальных багов и проверка удобства использования новых фич.
    - **Баг-трекинг:** Все найденные дефекты заносились в GitLab Issues и привязывались к соответствующим задачам.
