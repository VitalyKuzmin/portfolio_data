# Процесс разработки и CI/CD: Light in Night Pro

## 1. Философия
Гибридный CI/CD для сложного C++ desktop-приложения и сопутствующих backend-сервисов. Основная сложность — долгая кросс-компиляция C++ кода.

## 2. Стек инструментов
- **Репозиторий:** GitLab.
- **CI/CD:** **Jenkins** (для тяжелых C++ сборок) + **GitLab CI** (для Python-сервисов и оркестрации).
- **Сборка:** CMake, MSVC (Windows), GCC (Linux).
- **Тестирование:** GTest (для C++), Pytest (для Python).
- **Артефакты:** Nexus Repository.

## 3. Модель ветвления
- **`develop`:** Основная ветка разработки.
- **`release/v1.2.0`:** Ветка для стабилизации релиза. Здесь происходит только исправление багов.
- **`main`:** Содержит только стабильные, выпущенные версии (синхронизируется с `release` после релиза).
- **`feature/*`:** Ветки для новых фич.

## 4. CI/CD Пайплайн

### C++ Desktop-приложение (Jenkins)
Пайплайн в Jenkins запускается по веб-хуку из GitLab при коммите в `develop` или `release`.

1.  **Checkout:** Клонирование репозитория.
2.  **Build (Parallel):**
    - **Windows Slave:** Сборка проекта с помощью MSVC.
    - **Linux Slave (Astra):** Сборка с помощью GCC.
    - Этот этап самый долгий (до 40 минут).
3.  **Test (Parallel):**
    - Запуск Unit-тестов (GTest) на обеих платформах.
4.  **Package:**
    - Создание инсталлятора (`.exe` через NSIS для Windows, `.deb` для Linux).
    - Артефакты загружаются в Nexus.
5.  **Deploy:**
    - Внутренние сборки автоматически публикуются на тестовый сервер.
    - Релизные сборки выкладываются на портал для скачивания вручную.

### Python Backend-сервисы (GitLab CI)
Пайплайн для сервисов (лицензирование, телеметрия) был стандартным:
`lint` -> `test` -> `build-docker` -> `deploy-to-staging`.

## 5. Тестирование
- **Unit-тесты:** Покрывали математическое ядро (Calculation Engine) и логику backend-сервисов.
- **UI-тесты:** Использовался фреймворк **Airtest** (на основе computer vision), который запускался на стенде с виртуалками (Windows/Linux) и "прокликивал" основные пользовательские сценарии. Это позволило автоматизировать регрессионное тестирование UI.
- **Ручное тестирование:** QA-инженер проводил полное тестирование по тест-кейсам перед каждым релизом.
