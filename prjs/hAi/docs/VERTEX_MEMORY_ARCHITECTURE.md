# Vertex Memory Architecture (Гибридная Архитектура Памяти)

## 1. Обзор

Архитектура памяти hAi ("Vertex Memory") представляет собой гибридную систему, объединяющую **векторный поиск** для персонализации и **структурированную суммаризацию** для управления контекстом диалога и активности.

Цель: Предоставить агенту максимально релевантный контекст (кто пользователь, что он делал, о чем говорили), не перегружая лимит токенов.

## 2. Компоненты Памяти

### 2.1. User Profile (Векторная Память)
**Технология:** LanceDB + OpenAI Embeddings.
**Назначение:** Хранение фактов о пользователе, его предпочтениях, стеке технологий и стиле общения.
**Работа:**
1.  **Индексация:** При старте (или по команде) файл `user_prefs.json` загружается, разбивается на записи, для каждой генерируется эмбеддинг, и они сохраняются в локальную базу LanceDB.
2.  **Поиск (Retrieval):** При каждом запросе пользователя текст запроса превращается в вектор. В базе ищутся топ-K (по умолчанию 3) наиболее близких по смыслу фактов.
3.  **Контекст:** Найденные факты ("User prefers Python", "Be concise") добавляются в системный промпт в секцию `## User Profile Preferences`.

### 2.2. Episodic Memory (Эпизодическая Память / Summaries)
**Технология:** SQLite + LLM Summarizer.
**Назначение:** Хранение сжатой истории диалогов и активности пользователя.
**Работа:**
1.  **Chunking:** Старые сообщения и события активности группируются в "куски" (chunks).
2.  **Summarization:** Фоновый агент создает текстовое саммари для каждого куска.
3.  **Rolling Context:** В промпт попадает "комбинированное" саммари (объединение последних N кусков), обеспечивая непрерывность контекста без хранения всего текста.

### 2.3. Short-Term Memory (Краткосрочная Память / Raw Window)
**Технология:** In-Memory / SQLite query.
**Назначение:** Точное хранение последних реплик и действий для немедленной реакции.
**Работа:**
1.  В промпт всегда попадают последние N сообщений (Raw Chat) и событий (Raw Activity) в "сыром" виде.
2.  Это позволяет агенту помнить точные формулировки и детали недавнего контекста.

### 2.4. Long-Term Memory (Долгосрочная Память / Tools)
**Технология:** Инструменты (`tools`) + SQL Search.
**Назначение:** Доступ к глубокой истории, которая ушла из контекста.
**Работа:**
1.  Агент использует инструмент `recall_memory`, который объединяет функции поиска (`search_conversation`, `search_activity`).
2.  Поиск выполняется по ключевым словам или временным диапазонам.

## 3. Пайплайн Обработки Запроса (Request Pipeline)

Когда пользователь отправляет сообщение, система выполняет следующие шаги:

1.  **Векторный Поиск (User Profile):**
    *   *Вход:* Текст сообщения пользователя.
    *   *Действие:* `UserProfileManager` делает запрос к LanceDB.
    *   *Выход:* Список релевантных строк (например, "User prefers code examples in Python").

2.  **Сбор Контекста (Context Assembly):**
    *   **Global:** Загружается системный промпт, текущее время, ОС.
    *   **Profile:** Вставляются найденные предпочтения (Dynamic Context).
    *   **Summaries:** Из кэша/БД достается последнее комбинированное саммари диалога и активности.
    *   **Short-Term:** Из БД достаются последние сообщения и недавние события ActivityWatch.
    *   **Sensory:** Делается скриншот (если нужно) и добавляется в контекст.

3.  **Генерация Промпта:**
    *   Все эти данные собираются в единый структурированный промпт для LLM.

4.  **Выполнение (Inference):**
    *   LLM получает промпт, "помнит" предпочтения пользователя и контекст беседы, и генерирует ответ.

## 4. Структура Данных

### User Preferences (`user_prefs.json` -> LanceDB)
Файл в корне `Backend/`, который редактируется вручную (пока).
```json
[
  {
    "id": "pref_tech",
    "title": "Tech Stack",
    "text": "User works with Python, React and Electron.",
    "tags": ["tech"]
  },
  {
    "id": "pref_style",
    "title": "Style",
    "text": "Be concise and professional.",
    "tags": ["style"]
  }
]
```

### LanceDB (`Backend/data/lancedb_user_profile`)
Локальная папка с векторным индексом. Создается автоматически при первом запуске.
