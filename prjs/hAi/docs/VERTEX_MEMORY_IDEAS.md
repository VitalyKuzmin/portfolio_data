
## 1. Базовая идея памяти в hAi

**hAi = LLM + локальная embedding-модель + векторная память.**

- Одна **локальная embedding-модель** (небольшая, работает на любом десктопе).
- Одна **векторная БД** (Chroma / Qdrant / Milvus — не важно, концептуально).
- Все важные “кусочки жизни” пользователя (диалоги, действия, факты, знания)  
    превращаются в эмбеддинги и кладутся в разные **слои памяти**.
- Поиск по памяти = **семантический поиск** (по смыслу, а не по строкам).
- Большая LLM используется только в конце, чтобы красиво сформулировать ответ.
    

---

## 2. Компоненты архитектуры

### 2.1. Модели

- **Embedding-модель (локальная)**
    
    - Одна на всю систему.
        
    - Отвечает за: _текст → вектор смыслов_.
        
    - Используется всеми слоями памяти: диалоги, наблюдения, факты, знания.
        
- **LLM-слой**
    
    - **Маленькая модель / router**:
        
        - определяет intent (вопрос к памяти / к знаниям / просто обычный запрос),
            
        - решает, какие типы памяти дёргать.
            
    - **Основная LLM**:
        
        - получает уже собранный контекст (сырой хвост, summaries, найденные куски памяти, факты о пользователе),
            
        - генерирует финальный ответ.
            

---

### 2.2. Векторная БД

Одна система хранения, внутри — **коллекции / типы памяти**, различаются через `memory_type`/`collection` и метаданные.

Каждая запись в векторке:

```json
{
  "id": "...",
  "embedding": [...],
  "text": "...",         // кусок диалога / summary / факт / фрагмент документа
  "metadata": {
    "memory_type": "...",
    "timestamp": "...",
    "source": "...",
    "project": "...",
    "...": "..."
  }
}
```

Поиск:  
`query → embedding → vector search (+ фильтры) → top_k + metadata`.

## 3. Типы памяти (слои/коллекции)

### 3.1. Dialog memory

**a) `dialog_raw`**

- Что хранится:
    
    - куски реальной переписки (чанки по нескольку реплик),
        
    - метаданные: время, тема, канал (текст/голос).
        
- Для чего:
    
    - “о чём мы говорили про доение коров?”
        
    - “напомни, как мы настраивали FRP в прошлый раз?”
        

**b) `dialog_summaries` (эпизодическая память)**

- Саммари больших кусков диалога:
    
    - сессия, день, обсуждение проекта, “вечер про RAG”.
        
- Для чего:
    
    - “подведи итоги прошлой недели”
        
    - “расскажи коротко, что мы решили по архитектуре памяти hAi”
        

---

### 3.2. Observation memory (действия пользователя)

**a) `actions_raw`**

- Что хранится:
    
    - события: открыть файл, редактировать, смотреть видео, запускать скрипт, переключить окно.
        
    - метаданные: путь, url, приложение, timestamp, тип действия.
        
- Для чего:
    
    - “что я делала вчера вечером?”
        
    - “какой файл я редактировала перед этим багом?”
        
    - “что я смотрела на YouTube вчера?”
        

**b) `actions_summaries`**

- Саммари активности за день/сессию:
    
    - “утром работали над hAi, вечером — Япония и поездка”.
        
- Для чего:
    
    - “дай обзор, чем я занималась вчера”
        
    - “что мы делали на прошлых выходных?”
        

---

### 3.3. User profile / персональный портрет

**`user_profile`**

- Краткие факты о пользователе:
    
    - стили ответов (коротко/подробно/с эмодзи),
        
    - языковые предпочтения,
        
    - любимые инструменты (VSCode, Figma, Obsidian),
        
    - поведенческие паттерны (ночной режим работы, не любит жёсткие диеты и т.п.).
        
- Использование:
    
    - почти при каждом запросе:
        
        - запрос → embedding → top_k фактов о пользователе,
            
        - в промт попадают 1–5 релевантных фактов, а не весь профиль.
            
- Это позволяет:
    
    - не пихать “все настройки юзера” в каждый промт,
        
    - а подмешивать только то, что **по смыслу относится к текущему вопросу**.
        

---

### 3.4. Knowledge vaults (пользовательская БД знаний)

**`knowledge_*` (несколько vault’ов)**

- Пользователь может создать “хранилище знаний”:
    
    - `vault_hAi_docs`
        
    - `vault_book`
        
    - `vault_project_X`
        
- Туда загружаются:
    
    - заметки, документы, pdf, конспекты, главы книги, документация.
        
- Внутри:
    
    - документы режутся на чанки → эмбеддинги → векторка с метаданными.
        
- Для чего:
    
    - “ответь по моей документации по hAi”
        
    - “какая структура у моей книги, напомни про вторую главу”
        
    - “какие проблемы с музейным освещением я уже описала?”
        

Можно включать режим:

> “отвечай только на основе этого vault, не фантазируй”.

---

### 3.5. System map (карта компьютера)

Это отдельный слой, **частично векторный, частично структурный**.

- **Структурный индекс (без эмбеддингов):**
    
    - дерево пользовательских папок,
        
    - список приложений,
        
    - типы файлов, пути, расширения, время изменения.
        
    - хранится в SQLite/JSON, не во векторке.
        
- **Семантический индекс файлов (векторка):**
    
    - только для “умных” файлов (md, txt, docx, pdf, код),
        
    - эмбеддинг по названию + началу содержимого или summary,
        
    - метаданные: path, project, tags.
        
- Для чего:
    
    - “найди файл с планом по Японии”
        
    - “найди заметку про коров”
        
    - “найди презентацию по hAi, с которой я работала в понедельник”.
        

---

### 3.6. Tasks / intents (опционально, но круто)

**`tasks_intents`**

- Хранит незакрытые задачи и идеи:
    - “переписать watcher на Rust”
    - “исследовать embedding-модели”
    - “сделать UX прототип голосового режима”.
- Использование:
    - “какие задачи у нас висят по hAi?”
    - “о чём я просила тебя напомнить, но мы не сделали?”

---

### 3.7. Episodic timeline (надстройка)

Можно строить “ленту жизни” поверх summaries:

- записи вида:
    - “2025-01-10 — обсуждали RAG и память в hAi”
    - “2025-01-11 — настраивали FRP и SSH”.
- Использование:
    - “что мы вообще делали в январе?”
    - “напомни день, когда мы обсуждали embedding-модели”.

## 4. Как работает запрос (общий pipeline)

Условный `on_user_message(msg)`:

1. **Обновление краткосрочной памяти**
    - добавить `msg` в `dialog_raw` (внутреннюю структуру short-term),
    - обновить/пересчитать rolling-summary по диалогу (если нужно),
    - сохранить последние наблюдения (actions_raw).
2. **Router / director**
    - понять intent:
        - это вопрос к знаниям? к памяти? просто чат?
        - нужно ли смотреть диалоги / действия / персональный профиль / vault?
3. **Запросы к памяти**
    - при необходимости:
        - `remember_from_dialog(msg)` → векторный поиск по `dialog_raw/summaries`,
        - `remember_actions(msg)` → поиск по `actions_*`,
        - `remember_profile(msg)` → поиск по `user_profile`,
        - `remember_knowledge(msg, active_vault)` → RAG по нужному vault,
        - запросы к `system_map` (структурный поиск по файлам и приложениям).
4. **Сбор контекста**
    - сырые последние реплики,
    - rolling-summary (недавний диалог),
    - релевантные куски памяти из разных коллекций,
    - релевантные факты о пользователе,
    - куски из knowledge-vault’ов (если надо).
5. **Вызов основной LLM**
    - LLM получает уже готовый, компактный контекст,
    - генерирует финальный ответ и/или план действий.

## 5. Ключевые принципы, которые мы зафиксировали

- **Одна embedding-модель** → общая карта смыслов для всех типов памяти.
- **Векторка = поиск, а не генерация**:
    - внутри `memory_search` нет LLM, только математика (similarity + фильтры).
- **Метаданные хранят точные штуки**:
    - URL, дата, путь к файлу, тип события.
    - Эмбеддинг хранит “смысл”.
- **Персональный профиль не пихается целиком в каждый промт**:
    - всегда можно дернуть `user_profile` по смыслу текущего вопроса.
- **Суммаризационная память не заменяется RAG, а дополняет его**:
    - rolling-summary всегда идёт в промт,
    - периодические summaries становятся эпизодической памятью (`dialog_summaries`, `actions_summaries`).
- **Чаты как “отдельные сессии” перестают быть важными**:
    - один общий мозг, который может вспомнить любой эпизод “из прошлого”, независимо от того, в каком это было чате/окне.
