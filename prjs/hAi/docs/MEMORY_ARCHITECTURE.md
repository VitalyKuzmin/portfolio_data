# Архитектура Памяти Агента

## 1. Обзор

Цель системы памяти — предоставить AI-агенту релевантный контекст для выполнения задач, одновременно управляя ограничениями по количеству токенов в промпте. Архитектура имитирует когнитивную модель человеческой памяти и состоит из трех основных компонентов:

1.  **Global Memory (Глобальная Память):** Статический контекст, который всегда присутствует в системном промпте.
2.  **Short-Term Memory (Краткосрочная Память):** Недавние события (диалог и активность), которые частично хранятся в "сыром" виде, а частично — в виде кратких саммари.
3.  **Long-Term Memory (Долгосрочная Память):** Вся остальная история, доступная агенту только через специальные инструменты.

---

## 2. Global Memory (Глобальная Память)

Это набор инструкций и данных, который формирует основу системного промпта и присутствует в каждом запросе к агенту.

### Компоненты:

*   **Global Prompt (Глобальный Промпт):** Основной набор инструкций, определяющий личность, цели и общие правила поведения агента.
    *   *Настройка:* `Settings -> Memory -> Global Memory -> Global Prompt`.
*   **System Data (Системные Данные):** Динамически добавляемая информация о текущем состоянии системы (ОС, дата и время, характеристики и т.д.).
    *   *Настройка:* `Settings -> Memory -> Global Memory -> System Data`.
*   **Rules (Правила):** Список пользовательских правил, которым агент должен следовать.
    *   *Настройка:* `Settings -> Memory -> Global Memory -> Rules`.

---

## 3. Short-Term Memory (Краткосрочная Память) и Суммаризация

Этот компонент отвечает за память о недавних событиях. Чтобы уместиться в лимит токенов, и история диалога (`Conversation History`), и история активности пользователя (`User Activity History`) используют идентичный механизм **"кусочной" (chunk-based) суммаризации**. Старая информация сжимается в саммари, освобождая место для новых данных.

### 3.1. Компоненты в Промпте Агента

Для обоих типов истории в промпт агента попадают два основных компонента:

1.  **Raw Window (Окно "сырых" данных):** Самые последние события (сообщения или действия пользователя), которые передаются в промпт в "сыром" виде.
    *   *Особенность для User Activity:* Фактический размер этого окна в промпте равен `Raw Activity Window` минус `Summarization Trigger %`, чтобы освободить место для буфера "на суммаризация".
2.  **Summary (Саммари):** Краткое саммари более старых событий. **Важно:** в промпт попадает только **одно, самое последнее "комбинированное" саммари**, которое кэшируется в памяти после фоновой задачи суммирования.

### 3.2. Настройки

Настройки для обоих типов памяти полностью аналогичны и находятся в соответствующих разделах:
*   `Settings -> Memory -> Memory Stores -> Conversation History`
*   `Settings -> Memory -> Memory Stores -> User Activity History`

Ключевые настройки включают:
*   `Raw ... Window`: Лимит токенов для "сырых" данных.
*   `Summarization Trigger`: Порог (в % от `Raw Window`). Когда буфер данных "на суммаризацию" превышает этот порог, запускается фоновая задача.
*   `Summary Target Size`: Целевой размер (в токенах) для **выходного** саммари, т.е. размер одного "куска".
*   `Combine Summaries`: Количество **старых** "чистых" саммари из базы данных, которые будут объединены с новыми данными для создания "комбинированного" саммари для промпта. Также это значение определяет, сколько "чистых" саммари хранится в базе данных.
*   `Enable Summarization`: Включает/выключает функцию.
*   `Summarizer Agent`: Нейрон, используемый для суммаризации.

### 3.3. Процесс Суммаризации (в фоновом режиме)

Процесс абсолютно идентичен для диалога и активности и разделен на два потока для создания двух типов саммари:

1.  **Создание "Чистого" саммари для Базы Данных:**
    *   **Триггер:** Процесс запускается, когда объем данных в буфере "To Be Summarized" (данные старше `Raw Window`, но новее последнего саммари) превышает `Summarization Trigger`.
    *   **LLM-вызов:** Система берет **только** данные из буфера "To Be Summarized" и просит `Summarizer Agent` создать из них "чистое" саммари размером не более `Summary Target Size`.
    *   **Сохранение:** Это "чистое" саммари сохраняется в базу данных как новый "кусок". Если общее количество "кусков" превышает значение `Combine Summaries`, самый старый "кусок" **удаляется**, поддерживая таким образом "скользящее окно" памяти.

2.  **Создание "Комбинированного" саммари для Промпта Агента:**
    *   **Условие:** Этот шаг выполняется, только если `Combine Summaries = N` больше нуля.
    *   **Сбор данных:** Система берет `N` последних "чистых" саммари из базы данных и объединяет их с **новыми данными** из буфера "To Be Summarized".
    *   **LLM-вызов:** Собранный контекст отправляется `Summarizer Agent` с просьбой создать новое, единое "комбинированное" саммари размером не более `Summary Target Size`.
    *   **Кэширование:** Это "комбинированное" саммари **не сохраняется в базу данных**, а временно кэшируется в памяти. Именно оно будет использовано в следующем промпте агента для обеспечения максимального контекста.

## 4. Long-Term Memory (Долгосрочная Память)

Вся информация, которая слишком стара, чтобы попасть в `Short-Term Memory` (включая старые "куски" саммари и старые "сырые" сообщения/события).

*   **Доступ:** Эта информация **не включается** в промпт автоматически. Агент должен "вспомнить" ее, используя специальные инструменты.
*   **Инструменты:**
    *   `recall_memory`: Динамический инструмент, который предоставляет доступ к функциям плагинов долгосрочной памяти. Основные функции:
        *   `conversation.search_conversation(text, start_time, end_time, limit)`: Поиск по истории диалога.
            *   **Ограничения:** Максимальный период поиска — **14 дней**. Максимальный размер ответа — ~16 000 токенов (возвращаются самые свежие результаты).
        *   `user_activity.search_activity(text, start_time, end_time, watchers, limit)`: Поиск по истории активности пользователя.
            *   **Ограничения:** Максимальный период поиска — **7 дней**. Максимальный размер ответа — ~16 000 токенов (возвращаются самые свежие результаты).
            *   **Параметры:** `start_time` и `end_time` задаются в формате ISO 8601. Список доступных `watchers` передается в промпт динамически.

## 5. Отображение в UI (Модальные окна "Chat History" и "Activity History")

Интерфейс истории визуализирует состояние памяти:

*   **Raw Window (Newest):** Самые свежие элементы, которые попадают в промпт в "сыром" виде.
*   **To Be Summarized:** Элементы, ожидающие суммаризации.
*   **Summarized History (Oldest):** Эleменты, покрытые последними `N` "кусками" саммари (где `N` — значение `Combine Summaries`).
*   **Long-Term Memory:** Вся остальная, более старая история. Эти данные **не отображаются** в модальном окне, чтобы не перегружать интерфейс, и доступны агенту только через инструменты.
